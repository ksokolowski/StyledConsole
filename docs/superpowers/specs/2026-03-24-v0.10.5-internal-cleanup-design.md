# v0.10.5 Internal Cleanup — Design Spec

**Date:** 2026-03-24
**Version target:** 0.10.5 (patch release)
**Constraint:** No public API changes. All fixes are internal-only.

## Background

A baseline code review of StyledConsole v0.10.4 identified 2 critical, 5 important, and 7 suggestion-level issues. This spec covers the subset appropriate for a patch release: internal cleanups and a partial fix for the most impactful critical issue.

## Scope

### In scope (8 commits, ordered lowest-to-highest risk)

1. **Move benchmark/utility scripts out of `tests/`**
2. **Extract exception classes to `exceptions.py`**
3. **Library-level NullHandler for logging**
4. **Add logging to bare `except Exception` blocks**
5. **Simplify `EffectSpec.with_*()` with `dataclasses.replace()`**
6. **Consolidate duplicate Renderer protocols**
7. **Scope render target ContextVar at render-time + document thread-safety**
8. **Version bump to 0.10.5**

### Deferred to v0.11.0+

- Trim `__init__.py` public API surface (110+ exports)
- Reduce `Console.frame()` parameter count (17 params)
- Extract `StyleResolver` from Console class (1853 lines)
- Scope icon mode and Rich patching per-Console instance
- Remove deprecated legacy gradient parameters

## Commit Details

### Commit 1: Move benchmark/utility scripts out of `tests/`

**Risk:** None
**Files changed:**
- `tests/benchmark_visual_width.py` → `benchmarks/benchmark_visual_width.py`
- `tests/verify_warnings.py` → `scripts/verify_warnings.py`

**Rationale:** These are utility/benchmark scripts, not tests. They sit alongside test files and could cause confusion. Moving them keeps the test directory focused.

### Commit 2: Extract exception classes to `exceptions.py`

**Risk:** Very low
**Files changed:**
- Create `src/styledconsole/exceptions.py` — define `StyledConsoleError`, `RenderError`, `ExportError`, `TerminalError`
- `src/styledconsole/__init__.py` — remove inline definitions, re-export from `exceptions.py`
- Any internal files that import these exceptions — update import paths

**Rationale:** Exception classes are currently defined inline in `__init__.py`. A dedicated module keeps `__init__.py` focused on imports and follows standard library patterns.

**Public API:** Unchanged. `from styledconsole import RenderError` continues to work.

**Verification:** `uv run pytest` — all tests pass. Grep for any direct imports of exceptions from `__init__`.

### Commit 3: Library-level NullHandler for logging

**Risk:** Very low
**Files changed:**
- `src/styledconsole/__init__.py` — add `logging.getLogger("styledconsole").addHandler(logging.NullHandler())`
- `src/styledconsole/core/rendering_engine.py` — remove `if not logger.handlers:` block and `StreamHandler` setup (~line 102)
- `src/styledconsole/core/export_manager.py` — remove `if not logger.handlers:` block (~line 60)

**Rationale:** Per Python logging best practices, libraries should add a NullHandler at the package level and let applications configure logging. The current per-module handler setup can create duplicate handlers when loggers propagate.

**Verification:** `uv run pytest` — all tests pass. No visible output changes (handlers were only active at DEBUG level).

### Commit 4: Add logging to bare `except Exception` blocks

**Risk:** Very low
**Files changed:**
- `src/styledconsole/export/emoji_renderer.py` — 6 occurrences: add `logger.debug("...", exc_info=True)`
- `src/styledconsole/core/rendering_engine.py` — ~line 621: add debug logging
- `src/styledconsole/export/image_exporter.py` — ~line 473: add debug logging
- Where the expected exception type is obvious (e.g. `OSError` for file/font operations), narrow the `except` clause

**Rationale:** Bare `except Exception` blocks that silently discard errors make debugging very difficult. Adding debug-level logging preserves graceful degradation while making failures diagnosable.

**Verification:** `uv run pytest` — all tests pass. Manual: set `logging.basicConfig(level=logging.DEBUG)` and trigger a failure path to confirm logging works.

### Commit 5: Simplify `EffectSpec.with_*()` methods

**Risk:** Low
**Files changed:**
- `src/styledconsole/effects/spec.py` — replace `with_direction()`, `with_target()`, `reversed()`, `with_phase()` method bodies

**Before (each method, ~10 lines):**
```python
def with_direction(self, direction: str) -> "EffectSpec":
    return EffectSpec(
        colors=self.colors,
        direction=direction,
        target=self.target,
        # ... 8 more fields
    )
```

**After (each method, 1 line):**
```python
def with_direction(self, direction: str) -> "EffectSpec":
    return dataclasses.replace(self, direction=direction)
```

**Rationale:** The current implementation manually copies all 11 fields. Adding a new field requires updating all 4 methods. `dataclasses.replace()` is the idiomatic solution and eliminates this maintenance burden.

**Verification:** `uv run pytest tests/unit/test_effects.py` — existing tests cover all `with_*()` methods.

### Commit 6: Consolidate duplicate Renderer protocols

**Risk:** Low
**Files changed:**
- `src/styledconsole/types.py` — assess `Renderer` protocol usage
- `src/styledconsole/rendering/base.py` — canonical `Renderer` protocol stays here
- `src/styledconsole/__init__.py` — update re-exports if needed
- Internal files importing either protocol — consolidate to one

**Approach:**
1. Check if `types.Renderer` is used anywhere internally or by tests
2. If unused: remove it from `types.py`
3. If used: rename to `LegacyRenderer`, add deprecation comment, point to `rendering.base.Renderer`
4. Ensure `__init__.py` exports are unchanged (the name `Renderer` still resolves)

**Rationale:** Two different `Renderer` protocols with different signatures creates type confusion. The `rendering/base.py` version is the v0.10.0 API; the `types.py` version appears to be a legacy holdover.

**Verification:** `uv run pytest` — all tests pass. `mypy` or `pyright` check if configured.

### Commit 7: Scope render target ContextVar at render-time

**Risk:** Medium (highest in this release)
**Files changed:**
- `src/styledconsole/console.py` — remove `set_render_target()` from `__init__`, add context manager usage at render call sites (`frame()`, `text()`, `banner()`, `table()`, etc.)
- `src/styledconsole/utils/text.py` — verify `_render_target_context` ContextVar is correctly read
- Create `docs/THREAD_SAFETY.md` — document what is and isn't thread-safe

**Approach:**
1. Create a private context manager (e.g. `_render_context(target)`) that sets the ContextVar on entry, resets on exit
2. Wrap each public render method's body with this context manager
3. Remove the `set_render_target()` call from `Console.__init__`
4. The `_patch_rich_cell_len_for_export()` and `set_icon_mode()` calls in `__init__` remain as-is (documented as global singletons, deferred to v0.11.0)

**Thread-safety documentation (`docs/THREAD_SAFETY.md`):**
- Render target: per-call safe (v0.10.5+)
- Icon mode: global singleton, not thread-safe (planned for v0.11.0)
- Rich cell_len patching: global, not thread-safe (planned for v0.11.0)
- Recommendation: use one Console per thread for now

**Verification:**
- `uv run pytest` — all tests pass
- Manual: create two Console instances (terminal + HTML export), verify they don't interfere
- Verify examples still work: `cd ../StyledConsole-Examples && uv run python run_examples.py --auto --fast`

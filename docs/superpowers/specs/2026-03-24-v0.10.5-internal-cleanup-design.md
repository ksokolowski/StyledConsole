# v0.10.5 Internal Cleanup — Design Spec

**Date:** 2026-03-24
**Version target:** 0.10.5 (patch release)
**Constraint:** No public API changes. All fixes are internal-only.

## Background

A baseline code review of StyledConsole v0.10.4 identified 2 critical, 5 important, and 7 suggestion-level issues. This spec covers the subset appropriate for a patch release: internal cleanups and a partial fix for the most impactful critical issue.

## Scope

### In scope (8 commits, ordered lowest-to-highest risk)

1. **Move benchmark/utility scripts out of `tests/`**
2. **Extract exception classes to `exceptions.py`**
3. **Library-level NullHandler for logging**
4. **Add logging to bare `except Exception` blocks**
5. **Simplify `EffectSpec.with_*()` with `dataclasses.replace()`**
6. **Consolidate duplicate Renderer protocols**
7. **Scope render target ContextVar at render-time + document thread-safety**
8. **Version bump to 0.10.5**

### Deferred to v0.11.0+

- Trim `__init__.py` public API surface (110+ exports)
- Reduce `Console.frame()` parameter count (17 params)
- Extract `StyleResolver` from Console class (1853 lines)
- Scope icon mode and Rich patching per-Console instance
- Remove deprecated legacy gradient parameters

### Commit 3: Library-level NullHandler for logging

**Risk:** Very low
**Files changed:**
- `src/styledconsole/__init__.py` — add `logging.getLogger("styledconsole").addHandler(logging.NullHandler())`
- `src/styledconsole/core/rendering_engine.py` — remove `if not logger.handlers:` block and `StreamHandler` setup (~line 109)
- `src/styledconsole/core/export_manager.py` — remove `if not logger.handlers:` block (~line 71)
- `src/styledconsole/core/terminal_manager.py` — remove `if not logger.handlers:` block (~line 74)

**Behavioral note:** The current `debug=True` path on `Console.__init__` triggers these handler-setup blocks so debug output appears on stderr. With the per-module handlers removed, `debug=True` must be preserved by having `Console.__init__` add a `StreamHandler` to the `"styledconsole"` logger when `debug=True` is passed. This keeps debug behavior working while still following the NullHandler library pattern for non-debug usage.

**Rationale:** Per Python logging best practices, libraries should add a NullHandler at the package level and let applications configure logging. The current per-module handler setup can create duplicate handlers when loggers propagate.

**Verification:** `uv run pytest` — all tests pass. Manual: verify `Console(debug=True)` still produces debug output.

### Commit 4: Add logging to bare `except Exception` blocks

**Risk:** Very low
**Files changed:**
- `src/styledconsole/export/emoji_renderer.py` — 6 occurrences: add `logger.debug("...", exc_info=True)`
- `src/styledconsole/core/rendering_engine.py` — ~line 621: add debug logging
- `src/styledconsole/export/image_exporter.py` — ~line 473: add debug logging
- Where the expected exception type is obvious (e.g. `OSError` for file/font operations), narrow the `except` clause

**Rationale:** Bare `except Exception` blocks that silently discard errors make debugging very difficult. Adding debug-level logging preserves graceful degradation while making failures diagnosable.

**Verification:** `uv run pytest` — all tests pass. Manual: set `logging.basicConfig(level=logging.DEBUG)` and trigger a failure path to confirm logging works.

### Commit 5: Simplify `EffectSpec.with_*()` methods

**Risk:** Low
**Files changed:**
- `src/styledconsole/effects/spec.py` — replace `with_direction()`, `with_target()`, `reversed()`, `with_phase()` method bodies

**Before (each method, ~10 lines):**
```python
def with_direction(self, direction: str) -> "EffectSpec":
    return EffectSpec(
        colors=self.colors,
        direction=direction,
        target=self.target,
        # ... 8 more fields
    )
```

**After (each method, 1 line):**
```python
def with_direction(self, direction: str) -> "EffectSpec":
    return dataclasses.replace(self, direction=direction)
```

**Import note:** Add `import dataclasses` at the top of `spec.py` (currently only has `from dataclasses import dataclass, field`).

**Rationale:** The current implementation manually copies all 11 fields. Adding a new field requires updating all 4 methods. `dataclasses.replace()` is the idiomatic solution and eliminates this maintenance burden.

**Verification:** `uv run pytest tests/unit/test_effects.py` — existing tests cover all `with_*()` methods.

### Commit 6: Document and clarify duplicate Renderer protocols

**Risk:** Very low
**Files changed:**
- `src/styledconsole/types.py` — add clarifying docstring to `Renderer` protocol
- `src/styledconsole/rendering/base.py` — rename `Renderer` to `ObjectRenderer` (internal name)
- `src/styledconsole/rendering/__init__.py` — update export to use `ObjectRenderer`
- `src/styledconsole/declarative/facade.py` — update `TYPE_CHECKING` import to `ObjectRenderer`

**Key constraint:** `from styledconsole import Renderer` currently resolves to `types.py:Renderer` (signature: `render(content, **kwargs) -> list[str]`). This is the **public API** and must not change. The `rendering/base.py` `Renderer` is an internal protocol used only by `declarative/facade.py` under `TYPE_CHECKING`.

**Approach:**
1. Rename `rendering/base.py:Renderer` → `ObjectRenderer` to eliminate the name collision
2. Update the one internal reference in `declarative/facade.py`
3. Add a docstring to `types.py:Renderer` clarifying it is the public protocol
4. `__init__.py` export unchanged — `Renderer` still points to `types.py`
5. Full consolidation (merging the protocols or deprecating one) deferred to v0.11.0

**Rationale:** Two protocols with the same name but different signatures is confusing. Renaming the internal one eliminates ambiguity without touching the public API.

**Verification:** `uv run pytest` — all tests pass. Grep: confirm no remaining imports of `Renderer` from `rendering.base`.

### Commit 7: Scope render target ContextVar at render-time

**Risk:** Medium (highest in this release)
**Files changed:**
- `src/styledconsole/console.py` — remove `set_render_target()` from `__init__`, add context manager at render call sites
- `src/styledconsole/effects/__init__.py` — remove save/restore workaround in `_create_console_for_effects()` (~lines 127-129) that is no longer needed
- `src/styledconsole/utils/text.py` — verify `_render_target_context` ContextVar is correctly read; `set_render_target()` remains as public/semi-public API for external callers
- Create `docs/THREAD_SAFETY.md` — document what is and isn't thread-safe

**Approach:**
1. Create a private context manager (e.g. `_render_context(target)`) that uses `ContextVar.set()` and `ContextVar.reset(token)` — must use the Token returned by `.set()` for correct nesting, never reset to a hardcoded default
2. Wrap each public render method's body with this context manager, using `self._render_target` (stored at init from the `target` param) as the value
3. Remove the `set_render_target()` call from `Console.__init__`
4. Remove the save/restore workaround in `effects/__init__.py:_create_console_for_effects()` — this was working around the init-time side effect that no longer exists
5. Keep `set_render_target()` as a public function for external callers (e.g. `scripts/readme/examples.py` calls it at module level for image-mode width calculations)
6. The `_patch_rich_cell_len_for_export()` and `set_icon_mode()` calls in `__init__` remain as-is (documented as global singletons, deferred to v0.11.0)

**Default behavior outside render methods:** When `visual_width()` is called outside a render context manager (e.g. by user code or standalone effect functions), the ContextVar retains its default value `"terminal"`. This is the correct and expected behavior.

**Thread-safety documentation (`docs/THREAD_SAFETY.md`):**
- Render target: per-call safe via ContextVar (v0.10.5+)
- `set_render_target()` remains available for module-level overrides (e.g. image export scripts)
- Icon mode: global singleton, not thread-safe (planned for v0.11.0)
- Rich cell_len patching: global, not thread-safe (planned for v0.11.0)
- Recommendation: use one Console per thread for now

**Verification:**
- `uv run pytest` — all tests pass
- Manual: create two Console instances (terminal + HTML export), verify they don't interfere
- Verify examples still work: `cd ../StyledConsole-Examples && uv run python run_examples.py --auto --fast`
